GCP DATA ENGINEERING - MASTER REVISION (DETAILED EXPLANATION)
============================================================

--------------------------------------------------
SECTION 1: REAL GCP ARCHITECTURE (DETAILED)
--------------------------------------------------

1. OVERVIEW

In real-world data engineering, we build a system where data flows from
source systems (apps, logs, APIs) to analytics tools (dashboards).

This entire journey is called a DATA PIPELINE.

--------------------------------------------------

2. STEP-BY-STEP ARCHITECTURE EXPLANATION

STEP 1: DATA SOURCE
- This is where data is generated.
- Examples:
  - User clicks in app
  - Payment transactions
  - Logs from servers

👉 Think: "Where data is born"

--------------------------------------------------

STEP 2: PUB/SUB (INGESTION LAYER)

- Pub/Sub is used to collect data in real time.
- It acts like a message queue.

How it works:
- Producer sends data → Pub/Sub stores messages
- Consumers read data later

Why we use it:
- Decouples systems
- Handles high volume
- Supports real-time pipelines

👉 Think: "Temporary inbox for data"

--------------------------------------------------

STEP 3: DATAFLOW (STREAM PROCESSING)

- Dataflow processes data in real time.
- Built using Apache Beam.

What it does:
- Cleans data
- Filters unwanted records
- Transforms data

Example:
Raw click → extract user_id → clean → format

👉 Think: "Live data cleaner"

--------------------------------------------------

STEP 4: GCS (DATA LAKE)

- GCS stores raw and processed data.
- It is cheap and scalable.

Why we use:
- Store raw data for backup
- Store intermediate data

Layers:
- Raw → original data
- Processed → cleaned data

👉 Think: "Storage warehouse for data"

--------------------------------------------------

STEP 5: DATAPROC (BATCH PROCESSING)

- Dataproc runs Spark jobs.
- Used for large-scale batch processing.

What it does:
- Aggregation
- Complex transformations
- Joins large datasets

Example:
Daily sales aggregation

👉 Think: "Heavy-duty data processing engine"

--------------------------------------------------

STEP 6: BIGQUERY (DATA WAREHOUSE)

- BigQuery stores structured data for analytics.

What it does:
- Fast SQL queries
- Data analysis

Example:
SELECT total_sales FROM table

👉 Think: "Analytics engine"

--------------------------------------------------

STEP 7: LOOKER / BI TOOLS

- Used to visualize data.

What it does:
- Dashboards
- Reports
- Charts

👉 Think: "Business view of data"

--------------------------------------------------

STEP 8: CLOUD COMPOSER (ORCHESTRATION)

- Composer controls entire workflow.

What it does:
- Schedules jobs
- Manages dependencies
- Handles retries

Example:
Run Dataproc → then BigQuery → then email

👉 Think: "Manager of the pipeline"

SECTION 3: SCENARIO-BASED QUESTIONS (DETAILED)
--------------------------------------------------

1. DATA DELAY ISSUE

Problem:
Data not arriving on time

Solution:
- Use Sensors in Airflow
- Wait until data arrives before processing

--------------------------------------------------

2. PIPELINE FAILURE

Problem:
Job fails during execution

Solution:
- Use retries
- Send alerts
- Check logs

--------------------------------------------------

3. LARGE DATA PROCESSING

Problem:
Huge data size

Solution:
- Use Dataproc (Spark)
- Use distributed computing

--------------------------------------------------

4. REAL-TIME PROCESSING

Problem:
Need instant results

Solution:
- Use Pub/Sub + Dataflow

--------------------------------------------------

5. COST OPTIMIZATION

Problem:
High cloud cost

Solution:
- Use serverless tools
- Delete unused clusters
- Use partitioning

--------------------------------------------------

6. DATA QUALITY ISSUE

Problem:
Incorrect data

Solution:
- Add validation checks
- Filter bad data

--------------------------------------------------

7. DEPENDENCIES

Problem:
Tasks depend on each other

Solution:
- Use Airflow DAG dependencies

--------------------------------------------------

8. SCALING ISSUE

Problem:
System slow

Solution:
- Increase workers
- Use autoscaling

--------------------------------------------------

9. SECURITY ISSUE

Problem:
Unauthorized access

Solution:
- IAM roles
- Encryption

--------------------------------------------------

10. MONITORING ISSUE

No visibility

- Use logs and monitoring tools

SECTION 4: DEEP INTERVIEW UNDERSTANDING
--------------------------------------------------

Why not cron jobs?

- Cron cannot handle dependencies
- No monitoring
- No retries

Airflow solves all these problems.

--------------------------------------------------

Why Dataproc vs BigQuery?

- Dataproc → heavy processing (Spark)
- BigQuery → querying

--------------------------------------------------

Batch vs Streaming:

Batch:
- Scheduled jobs
- Large data

Streaming:
- Real-time
- Continuous data

--------------------------------------------------

System Interaction:

Composer → triggers jobs
Dataproc/Dataflow → process data
BigQuery → stores results
